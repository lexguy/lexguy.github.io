这里仅分享 ES6 中的 Promise 的特性和用法，只浅显回答是什么和怎么用的问题。 如果需要深入了解原理，请参考另外一篇。

Promise 是什么

以前，对于 JavaScript 中传统的异步编程，一般会有两种解决方案：回调函数，发送事件。如果流程和数据传递比较复杂，回调可以说是相当冗长繁复了。而 Promise ，就是为了解决异步编程的问题，较之于以前的方案更加强大优雅。

本质上来说，Promise 是一个对象，持有异步操作的状态，用来表示异步操作的不同结果。遵循两个原则：

  • 对象只有（如下）三种状态，状态由操作的结果决定，且没有其他方式更改

    • pending 进行中
    • fulfilled 已完成
    • rejected 已失败

  • 状态改变只有两种情况：

    • 从 pending 变为 fulfilled
    • 从 pending 变为 rejected

可以简单理解为 Promise 中的异步操作，只能从进行到结束，而且一旦结束，结果就不会再变。不能暂停，不能取消，也不能改变已经产生的结果。

区别于事件监听机制的错过事件后就得不到结果的特性，如果 Promise 异步操作已经结束，状态已经改变，那在此之后的任何时候依然是可以通过对 Promise 添加回调函数获取到这个结果的。

如果 Promise 异步操作中，状态还处于 pending未改变时，外部对 Promise 的具体阶段是不可见的，也不能手动取消该该异步操作。

后面用resolved 指代 fulfilled (已成功) 状态，rejected 表示失败的状态。还要区别于使状态发生改变的 resolve 方法 reject 方法。

Promise 怎么用

知道了 Promise 会有三种状态，处理异步操作，那具体咋用呢？

resolve reject 传递异步操作结果

通常来说，resolve 用于传递异步操作成功的结果，而 reject用于传递失败时候的结果或者出现的异常。当调用 resolve 传递异步操作结果的时候，Promise 的状态就从 pending(进行中) 改变成了 fulfilled (已完成)，对应的，调用 reject 传递结果时，Promise 的状态从 pending (进行中) 变成了 resolved (已失败)。

不过这也只是约定俗成，你仍然可以通过 reject 传递成功的结果而 resolve 传递失败的结果，但是这样会影响到正常取值，错误及异常处理的正常流程

then() 方法指定回调

Promise 对象实例生成，执行对应的异步操作获取到结果后，可以用 then 方法指定不同状态的回调。

其他 Promise API 使用

上面是 Promise 的一些基础使用，已经可以进行简单的异步任务的处理，但是如果是多个异步任务或者其他复杂的情形，还需要了解其他 Promise API。

  • 2. 调用 catch 捕获

经过上面两种方式捕获到错误，信息仍然是相同的，但是此时不会报错，因为错误已经视作被获取处理。

如果异步操作结果失败，你也可以在 Promise 中 throw error, 这和reject 效果一样，会使 Promise 变为 rejected状态。抛出的错误可以被 then 方法的第二个回调或者 catch 捕获。

不过如果在 resolve 语句之后抛出错误，是无效的 。可以理解为这都是引起 Promise 状态改变的操作， resolve 之后状态已经改变为 resolved，不能再抛出异常变为 rejected。

Promise 对象的错误如果不被捕获，就是一直向后传递，直到被 then 方法的第二个回调或者catch 方法捕获到。同时一旦 Promise 错误已经被捕获，再在后面添加 then 方法或者 catch就捕获不到了。

上面的 catch 方法是捕获不到错误的，因为错误已经被 then 方法的第二个回调捕获。

一般来说，使用更多的是用 catch 来捕获错误。相比于 then 方法的第二个回调，catch 除了可以捕获到 Promise 异步操作的错误，也可以捕获到 then 方法第一个回调的错误。

catch 方法返回的还是一个 Promise 对象，且是resolved 状态的。后面可以继续正常调用 then 方法。此时如果catch方法没有捕获到错误，则会直接跳过 catch , 执行后面的then方法。

catch 方法还可以继续抛出错误，再由后面的 catch 方法捕获。

总结：

catch 和 then 方法的第二个回调都可以捕获错误。

Promise 对象的错误会向后传递直到被捕获，同一个错误一旦被捕获就不能再次被捕获。

常用的 promise.then().catch() 中， catch 可以捕获对应 then 中的错误，因而更常用。

catch 方法的返回值依然是 Promise，可以继续链式调用then和 catch。

finally( )

我们通过 then 的两个回调函数对 Promise 两种状态分别进行处理，有时候也会有一些公共的操作。比如，不管 Promise 状态成功或者失败，希望都能手动刷新一次页面，那么这个操作就应该放在 finally 中。

finally表示不管 Promise 最后的状态，在执行 then 或者 catch 的回调函数之后，都会执行 finally 中的回调函数。

其实，finally 只是对 then 方法的两个回调函数中的一些公共操作的一次抽取，避免了成功和失败情况的相同操作写两次。

另外，因为 finally 的回调函数是没有参数的，意味着finally中进行的操作只能是状态无关的，和 Promise 状态相关的操作是不能抽取到这里的。

race( )

同样是将多个 Promise 实例包装成一个新的 Promise 实例，race 则表现出竞争的关系。race 原意 ‘竞速’ ，表示多个异步过程最先执行完的就是最终结果。对于多个 Promise，只要有一个改变状态，Promise.race()的状态就跟着改变，不管是 resolved 还是 rejected 。所传参数和Promise.all()一样是多个 Promise 实例。

allSettled( )

Promise.allSettled 也是对多个 Promise 实例包装成一个新的 Promise 实例。但是会确保所有的实例都返回结果之后才会进行下一步的回调，不管这些 Promise 实例是 resolved 还会rejected。

Promise.allSettled() 返回的 Promise 实例总是 fulfilled 的， 该 Promise 的结果是多个 Promise 的实例的结果的数组。

输出的数组依然按照对应的传入的 Promise 的顺序：

数组的每个元素是对象，用 status 表示状态，只能是字符串的 re solved 或者 rejected，value 表示成功状态的值，reason 表示失败状态的值。

相比于 Promise.all, Promise.allSettled 的最大优势在于可以确保所有操作都结束，只有都结束时才会有下一步的操作。

Promise.allStelled() 会在所有 Promise 实例的异步操作结束后变为 fulfilled 状态，不管每个 Promise 结果是 resolved 还是 rejected 。

any( )

如果说 Promise.all 表现出多个 Promise 实例的和 且 的关系，那么 Promise.any 表现的就是 或 的关系。Promise.any 也是将多个 Promise 实例包装成一个 Promise 对象，但状态规则是这样：

  • 多个 Promise 实例只要有一个fulfilled，返回的包装实例的状态就会是fulfilled。
  • 只有所有的 Promise 实例都是 rejected 状态时，最终的结果才会是 rejected。

注意：该方法仍是处于实验提案阶段，支持性尚且不够，这里只做简单介绍。
